# 报团链接结构化会话状态

## 背景

`find_group_tour_links` 需要确定的 `route_id`，但原有对话策略只通过“我选”“就选”等文本标记猜测用户是否选过路线。该方式无法可靠处理数字回复、点名目的地或路线切换，也可能把“我选报团”误判为路线选择。

## 规则

- 从已审核路线目录构建会话状态，状态至少包含 `selected_route_id` 和 `transport_mode`。
- 用户直接说出已审核路线名称或检索词时，将其映射为确定的 `selected_route_id`。
- 用户回复数字选择路线时，根据上一条助手路线选项按编号映射到 `selected_route_id`。
- 用户回复交通方式名称或数字时，根据上一条助手交通选项映射为结构化交通方式。
- 助手遗漏数字前缀但仍按顺序展示“自驾、报团、公共交通”时，用户回复 `1/2/3` 仍应按展示顺序映射；同时兼容 `2）`、`2：` 等编号符号。
- 助手已经明确输出“已确认选择某路线”时，可从已审核路线目录恢复该路线 ID，避免界面历史裁剪或前序数字选项格式变化导致路线状态丢失。
- 只有当前轮明确选择“报团”且会话状态已有 `selected_route_id` 时，才要求调用 `find_group_tour_links`，并在内部策略中明确传入该 ID。
- 路线已确定后，用户选择“报团”或常见误写“抱团”时，必须在当前轮直接查询；不得再次确认路线、日期、交通方式或是否开始查询。
- 是否调用工具由对话策略负责判断；一旦已经发起 `find_group_tour_links` 并传入 `route_id`，工具不得再次校验或询问报团状态。工具只校验该 ID 是否对应已审核路线，然后直接查询公开活动链接。
- 当前轮选择“报团”但路线未确定时，不得调用工具，只提示先选定路线。
- 用户后续明确选择另一条路线时，使用新路线覆盖旧路线，避免历史选择污染。

## 验收

- 点名“巴朗山”后回复“报团”，可确定巴朗山路线 ID 并触发工具。
- 路线回复 `1`、交通方式回复 `2` 时，可按上一轮编号正确触发工具。
- 仅说“我选报团”但没有已选路线时，不触发工具。
- 先选青城后山、后改为巴朗山再报团时，使用巴朗山路线 ID。

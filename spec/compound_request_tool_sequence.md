# 复合请求的工具调用顺序

## 场景

用户可能在一句话中同时提供相对日期、目的地、交通方式和查询目标，例如：“周六想报团去爬巴朗山，不知道天气怎么样”。

## 串行流程

1. 先调用相对日期工具解析“周六”。
2. 日期解析成功后，使用工具返回的具体日期调用节假日工具。
3. 模型从用户原话提炼目的地名称，调用推荐工具；推荐工具使用 `group_tour_search_terms_json` 匹配路线。只有唯一命中一条已审核路线时，才得到确定路线 ID。
4. 使用推荐工具返回的确定路线 ID 查询天气；不得跳过路线推荐，直接使用自然语言目的地或预先猜测的路线 ID 查询天气。
5. 如用户选择报团，再使用同一个确定路线 ID 查询商团链接。
6. 最后整合已经成功返回的结果。

这些工具不得在日期尚未解析时并行调用。任一步失败时停止依赖该结果的后续调用，说明失败并请用户补充信息；不得自行推算日期、星期、日期类型或编造路线 ID。

Agent 执行层每轮最多执行模型生成的第一个工具调用；同一轮生成的其余工具调用不执行。首个工具返回后，再让模型基于结果生成下一次调用，从机制上保证依赖参数不是预先猜测的。

“周六”和“周日”按从参考日期起最近的对应星期解析；若参考日期当天就是该星期，则解析为当天。

## 路线引用

当前对话策略应把命中的路线名称和路线 ID 明确提供给模型。天气与报团工具只能使用该 ID，不得把“巴朗山”等自然语言目的地名称当作 `route_id`。
作为防御性措施，天气和报团工具收到唯一命中的路线名称或检索词时，应先规范化成库内路线 ID。

# 游侠客成都一日徒步路线采集与生成流水线

## 目标

使用一个命令从指定的游侠客筛选页按展示顺序发现候选，筛选出指定数量的成都出发一日徒步产品，再调用
`qwen3.7-max` 联网核验，生成完整路线数据并增量更新运行路线库。

## 行为

- `--count` 为必填正整数；分页抓取、进度显示、检查点和本次成功路线数量均使用该值。
- `--page-url` 只接受游侠客域名的 HTTPS 筛选页。第一页加载该地址，后续页必须从当前页面可见的“下一页”
  或“后一页”链接读取真实 `href`，不得根据页码猜测或拼接 URL；游侠客同域 HTTP 分页 href 必须先升级为 HTTPS。
  候选按页面展示顺序读取并按规范化详情 URL 去重；累计唯一候选达到 `--count` 后立即停止分页。没有下一页、
  页面无有效活动或与上一页活动重复时也停止分页。
- 用户提供的筛选页候选统一视为单日行程，不再根据标题或详情正文判断、确认或排除行程天数。
  标题和详情正文只需能够确认成都出发及徒步活动；非徒步和非成都出发的产品不计入数量。
- 详情页已提取事实优先于模型补全。模型不得覆盖网页非空值，不得拼接不同走法的数据或虚构来源。
- `qwen3.7-max` 使用思考模式和流式响应完成联网核验；流结束后由流水线解析完整 JSON，不发送模型不支持的
  `response_format` 结构化输出参数。
- 流式响应中的空 `choices`、用量统计和无正文结束事件属于合法控制事件，应跳过；服务端显式错误、畸形 JSON
  或最终没有正文时才判定调用失败。
- 第一阶段保存包含来源页面和目标数量的候选检查点；元数据不一致时拒绝复用，`--refresh-links` 强制重新发现。
- 第二阶段逐条保存已核验路线检查点；`--restart` 保留候选并从头核验。失败时不得修改正式运行数据。
- 模型输出必须完整包含 `route`、`costs` 和 `traffic`，并通过字段、枚举、来源、单日属性、路线逻辑以及
  `route.confidence > 0.8`、`traffic.confidence > 0.8` 校验。
- 本次路线按规范化 `source_url` 优先、`route.id` 次优先增量合并到 `data/sample_routes.json`：命中时原位替换，
  URL 命中时保留旧 ID，未命中时追加，未匹配旧路线不删除。身份冲突必须抛出含义明确的异常。
- 本流水线生成的数据统一设置 `reviewed=true`，可在应用同步后直接参与推荐。
- 合并后的完整数组经全量校验后，分别原子写入 `data/sample_routes_select.json` 和 `data/sample_routes.json`，
  两个文件内容必须完全一致。

## 入口

统一入口为：

```powershell
python -m hiking_chatbi.youxiake_route_pipeline --count 20
```

可选参数包括 `--page-url`、`--links-file`、`--checkpoint-file`、`--refresh-links` 和 `--restart`。

## 错误处理

- 验证码、近乎空白的详情、候选不足、模型连续校验失败或正式文件校验失败均停止流水线并输出中文异常。
- 单条已确认合格的产品若模型生成失败，不得跳过后继续取更靠后的产品，以免破坏“靠前 N 条”的语义。
- 所有正式文件只在本次 N 条路线全部完成且合并结果通过校验后更新。

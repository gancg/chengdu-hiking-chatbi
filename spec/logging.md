# 日志规范

## 目标

- 所有模块使用 Python 标准库 `logging`，不得使用 `print` 输出运行信息。
- 日志用于定位启动、请求、业务处理、外部服务和降级问题。
- 日志不得记录 API Key、认证头、完整用户输入或其他敏感信息。终端聊天生成的助手回答
  通过日志输出，以保留终端交互能力。

## 配置

| 配置项 | 默认值 | 说明 |
|---|---|---|
| `CHATBI_LOG_LEVEL` | `INFO` | 支持 `DEBUG`、`INFO`、`WARNING`、`ERROR`、`CRITICAL` |

默认格式包含时间、级别、logger 名称和消息。应用入口负责初始化根日志配置，
模块通过 `logging.getLogger(__name__)` 获取 logger。

## 日志级别

- `DEBUG`：缓存命中、候选路线过滤等调试细节。
- `INFO`：服务启动、导入完成、请求完成、推荐和天气评估完成。
- `WARNING`：非法请求、外部服务异常后的可恢复降级。
- `ERROR`：导致当前操作失败的未预期异常，必须保留异常堆栈。

## 模块要求

- CLI、HTTP API、WebUI 和终端聊天启动或停止时输出日志。
- HTTP API 记录请求方法、路径、状态码，不记录请求体。
- 路线导入、推荐、交通估算、天气估算记录操作结果和关键标识。
- 和风天气记录请求开始、缓存命中和降级异常，不记录凭据。
- 未启用 Provider 属于正常配置状态，仅记录 `DEBUG`；已启用 Provider 调用异常记录 `WARNING`。
- 已明确降级处理的异常必须使用 `logger.warning(..., exc_info=True)` 留下堆栈。

## 验收标准

- `hiking_chatbi` 目录中不存在 `print` 调用。
- 日志配置可通过 `CHATBI_LOG_LEVEL` 调整。
- API 请求、天气服务异常和交通服务异常均产生对应日志。
- 日志测试不得依赖真实外部服务。

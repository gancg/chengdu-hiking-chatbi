# 出发日期解析与有效性确认

## 背景

路线推荐、天气和交通查询都依赖准确且有效的出发时间。用户可能使用相对日期，
也可能写出与实际日期不一致的星期，或者选择已经过去的时间。因此必须先由程序
统一解析和校验，得到明确的未来出发时间后，才能调用后续工具。

## 日期解析

- 增加只读工具 `resolve_departure_date`，解析常见相对日期表达。
- 支持：今天、明天、后天、本周末、下周末，以及“周三”“本周三”“下周三”等周一至周日的完整表达。
- 中文星期表达按自然周解释，一周从星期一开始、星期日结束；“下周三”表示当前自然周下一周的星期三，不得理解为七天后再遇到的第二个星期三。
- 工具参数包含“本周日出发”等自然短句时，应提取其中唯一受支持的相对日期表达再解析。短句包含多个相对日期时不得自行选择，应抛出明确异常。
- “本周末”始终表示参考日期所在自然周的星期六和星期日。
- 当前日期上下文必须包含 ISO 日期、程序计算的星期，以及“后天”的日期和星期。
- 工具返回每个候选日期的星期、是否早于或等于参考日期、是否需要确认未来时刻、节假日状态和交通日期类型。
- Agent 遇到相对日期表达时必须先调用工具；相对日期、具体日期和星期名称均以程序结果为准，不得自行推算。

## 一致性与有效性校验

- 用户同时给出日期和星期但二者不一致时，应指出冲突并采用程序核验结果，不得沿用错误星期。
- 出发时间必须严格晚于当前本地时间，这是路线推荐、天气和交通查询的前置硬约束。
- 日期已经过去时，应引用用户填写的时间内容，明确说明其早于当前时间，并且只确认新的未来出发时间。
- 相对日期解析为今天，但用户没有提供可验证为未来的具体时刻时，也必须确认新的出发时间。
- 当天提供具体时刻时，应与当前本地时间比较；未来时刻可以继续，已经过去的时刻必须重新确认。
- 时间未通过校验前不得调用路线推荐、天气、交通或节假日工具，不得替用户顺延日期或使用默认时间。

## 多候选日期

- 多个候选日期会影响推荐结果时，应使用编号选项请用户选择。
- 已经过期的候选日期不得作为可选项提供给用户；没有有效候选日期时只确认新的未来时间。

## 验收

- 参考日期为 `2026-06-13` 时，“本周末”返回 `2026-06-13（星期六）` 和
  `2026-06-14（星期日）`。
- “下周末”返回下一自然周的星期六和星期日。
- 参考日期为 `2026-07-05（星期日）` 时，“下周三”返回 `2026-07-08（星期三）`，不得返回 `2026-07-15`。
- 参考时间为 `2026-06-18` 时，“后天”返回 `2026-06-20（星期六）`。
- 用户给出的日期与星期冲突时，回复指出冲突并使用程序核验的星期。
- 当前为星期六时，用户说“本周六想去徒步”，回复明确指出“本周六”不晚于当前时间，并确认新的未来出发时间，不调用后续工具。
- 用户选择当天未来具体时刻时，继续现有推荐流程。
- 不支持的表达返回明确异常，不得猜测。
